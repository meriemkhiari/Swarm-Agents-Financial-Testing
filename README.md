# Swarm Fraud Detection

Real-time transaction fraud detection using a multi-agent swarm architecture. Independent scoring agents evaluate different risk dimensions in parallel; a weighted aggregator fuses their verdicts into a final decision with an LLM-generated justification. A companion AI feedback-loop harness automatically diagnoses and proposes fixes for failing tests.

## Architecture

```
                    ┌─────────────────────┐
                    │   tx_events channel  │
                    └──────────┬──────────┘
        ┌──────────┬───────────┼───────────┬──────────┐
        v           v           v           v
   agent_amount agent_device agent_frequency agent_geolocation
        │           │           │           │
        └──────────┴───────────┴───────────┘
                    │
                    v
              agent_scores channel
                    │
                    v
            AggregatorService
     (weighted fusion + Ollama justification)
                    │
                    v
              decisions channel
```

Four independent agents subscribe to `tx_events` over Redis pub/sub, each score every transaction on a different dimension, and publish to `agent_scores`. `AggregatorService` correlates each transaction's four scores, computes a weighted fraud score, and publishes a decision with a natural-language justification generated by a local LLM.

## Components

### Shared Infrastructure

**`agents/base_agent.py` — `BaseSwarmAgent`**
Abstract base class implementing the Template Method pattern. Handles Redis connection with exponential backoff, subscribe/publish plumbing, per-message concurrency (bounded via `asyncio.Semaphore`), graceful shutdown, and malformed-payload isolation. Concrete agents implement only `compute_score(tx: TransactionPayload) -> float`.

### Scoring Agents

| Agent | Feature | Method | Port |
|---|---|---|---|
| `agent_amount` | `amount` | Isolation Forest — density-based outlier detection over transaction amount, percentile-normalized (1st/99th) rather than hard min/max to preserve tail resolution | 8001 |
| `agent_frequency` | `tx_per_hour` | Z-score of per-user transaction count in a rolling time window against the population mean/std, sigmoid-transformed and centered at 2σ | 8002 |
| `agent_geolocation` | `velocity_kmh` | Haversine distance between a user's last two transaction locations divided by elapsed time, compared against a physical plausibility ceiling (900 km/h) | 8003 |
| `agent_device` | `device_novelty` | Frequency-table lookup — inverse of historical transaction count for that device hash; unseen devices score maximum novelty | 8004 |

Each agent trains at startup from a PaySim CSV (`agent_amount`, `agent_frequency`) or accumulates state at runtime (`agent_geolocation`, `agent_device`) and exposes Prometheus metrics on its own port.

### Aggregator

**`aggregator/aggregator_service.py` — `AggregatorService`**
Subscribes to `agent_scores`, buffers per-transaction scores via `TransactionCollector` until all four agents report or a collection timeout elapses (whichever first), computes a weight-renormalized fused score:

$$S_{final} = \frac{\sum_i w_i \cdot s_i}{\sum_i w_i}$$

so a partial (timeout-triggered) set of scores is never silently deflated by treating missing agents as contributing zero. Calls a local Ollama model to generate a two-sentence justification for the decision, then publishes to `decisions`. Includes Redis reconnection with backoff, bounded `_finalized` tracking (`OrderedDict`, capped), and a reused `httpx.AsyncClient` to avoid per-transaction connection overhead.

### Frontend

**`frontend/SwarmFraudDetectionForm.jsx`**
React form submitting a transaction to `/analyze` and rendering the returned decision, final score, latency, and justification. Monochrome grey UI; red and green are reserved exclusively for `FRAUD`/`LEGITIMATE` outcomes (and the error state) so the decision itself is the only source of color signal.

### AI Feedback-Loop Test Harness

**`ai_feedback_loop/`**
A standalone tooling layer added on top of the existing pytest suite — it does not modify agent or aggregator behavior. It runs the test suite, and for every failure, routes it through a local, Ollama-backed AutoGen pipeline:

```
pytest --json-report
      │
      v
FailureRecord[] ──> query similar past defects (Chroma + Ollama embeddings)
      │
      v
failure_analyzer  ──>  fix_proposer  ──>  patch_critic  ──>  defect_filer
 (root cause)         (unified diff)     (APPROVE/REJECT)   (JSON record)
                                                │
                              APPROVE + auto_apply_patches=True
                                                │
                                                v
                                   patch applied (backed up first)
                                                │
                                                v
                                        re-run pytest, repeat
```

`auto_apply_patches` defaults to `False` — the harness produces filed defect records and proposed diffs under `.feedback_loop/defects/` for human review, without touching source. Every processed failure is indexed back into the vector store, so later failures retrieve relevant historical context automatically (RAG). The critic is instructed to reject any patch that loosens a test assertion instead of fixing the underlying defect, though this is an LLM instruction, not a hard constraint — treat auto-apply as appropriate for local iteration, not for changes headed to `main` without a human diff review.

## Setup

### Core services

```bash
docker compose up -d
```

Each agent and the aggregator read configuration from environment variables (`REDIS_URL`, channel names, `THRESHOLD_AGENT_*`, `PAYSIM_CSV_PATH`, `OLLAMA_MODEL`, `OLLAMA_BASE_URL`) with sane defaults for local development.

### Feedback-loop harness

```bash
ollama pull llama3
ollama pull nomic-embed-text
ollama serve
pip install -r requirements-feedback-loop.txt --break-system-packages
python run_feedback_loop.py
```

## Notable Fixes Since Initial Implementation

| Component | Original defect | Fix applied |
|---|---|---|
| `agent_amount` | Hard min/max normalization clipped and lost resolution at the tail | Switched to 1st/99th percentile bounds |
| `agent_device` | `OneClassSVM` trained on MD5-hashed device IDs — distance between hash values carries no statistical meaning | Replaced with a frequency-table lookup; novelty is inverse historical transaction count |
| `agent_frequency` | `deque(maxlen=100)` silently truncated burst counts above 100, masking the exact pattern the agent exists to detect; unbounded per-user dict growth | Unbounded deque with time-based eviction only; dict entries removed once a user's window fully drains |
| `agent_geolocation` | Same-instant duplicate events and out-of-order (redelivered) messages both scored as maximum fraud (`1.0`); unbounded per-user position tracking | Same-location/same-instant now scores `0.0`; negative elapsed time is logged and scored `0.0` without corrupting the true last-known position; `OrderedDict` LRU cap added |
| `BaseSwarmAgent` | Caught `json.JSONDecodeError` where Pydantic actually raises `ValidationError`, letting malformed payloads crash the loop; fully sequential message handling | Corrected exception type; per-message tasks now scheduled concurrently under a bounded `asyncio.Semaphore`; added graceful shutdown |
| `AggregatorService` | Weighted score used a fixed denominator of 1.0, so a timeout-triggered partial result silently deflated `S_final`; LLM justification call blocked the entire message loop; no reconnection logic; unbounded `_finalized` set | Weight sum now renormalized to received agents only; justification scheduled as a background task; Redis reconnection with backoff added; `_finalized` bounded via `OrderedDict` |
| Frontend | Purple/blue gradient accents throughout | Restricted to a grey palette; red and green reserved exclusively for the `FRAUD`/`LEGITIMATE` decision and the error state |

## Testing

Each agent and the aggregator has a dedicated pytest suite under `tests/unit/`, combining behavioral assertions (does the model actually detect what it's supposed to), regression guards (does a previously fixed defect stay fixed), degenerate-input edge cases, and Hypothesis-based property tests asserting the `[0.0, 1.0]` score contract holds universally. Run the full suite with:

```bash
pytest tests/unit -v
```

To have failures automatically analyzed and, optionally, patched:

```bash
python run_feedback_loop.py
```

See `README_feedback_loop.md` for the harness's full architecture and safety model.
